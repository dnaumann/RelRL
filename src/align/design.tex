% ============================================================
\section{About}
% ============================================================

\emph{bicommands} are product programs that pair up the execution steps of two
unary methods, together with annotations that serve as the proof skeleton for
relational verification. Manual construction of bicommands runs into two issues:

\begin{enumerate}
  \item it is tedious, error prone and time consuming; and
  \item an alignment well suited to the relational specification often takes a
        few iterations of tinkering to discover.
\end{enumerate}

\emph{Align mode} is a workflow within WhyRel that helps construct bimethods which are bicommands over unary methods. The steps are as follows:

\begin{enumerate}
  \item CLI invocation (\S\ref{sec:call}): Receives two unary methods along with a relational specification from command line.
  \item Default Alignment: Computes a \emph{sequential alignment}, the bicommand $(C \mid C')$, of two unary method bodies $C$, $C'$
  \item Serving: A local HTTP API (and a browser UI) for inspection; and
  read-only (text/HTML) or interactive (the API above).
  \item Optimizing: User manipulates the alignment using \emph{rewrite} rules (\S\ref{sec:rewrite})  and suggest  annotations. 
  \item Export: The bimethod can then be exported as a .rl text file.
\end{enumerate}

An overview of the codebase is in \S\ref{sec:code}.

% ============================================================
\section{Calling Align}
\label{sec:call}
% ============================================================

Align mode is invoked as:
\begin{lstlisting}
whyrel -align -l <LMod> <LMeth> -r <RMod> <RMeth> files...
       [-o <file>] [-i] [-port <n>]
       [-rpre <rformula>] [-rpost <rformula>]
\end{lstlisting}
    
\texttt{main.ml} parses and type-checks the input files, then dispatches to
align mode instead of the translation pipeline.

\begin{description}
  \item[\code{-l} / \code{-r}] left/right module and method to align.
  \item[\code{-o <file>}] output file for export (default
        \code{<stem>_align_output.rl}).
  \item[\code{-i}] start the \emph{interactive} rewriting server (\S\ref{sec:interactive}).
  \item[\code{-port <n>}] port for any of the servers (default \code{8080}).
  \item[\code{-rpre} / \code{-rpost}] take single relational formulas in WhyRel syntax 
      (e.g.\ \code{"Agree x"}, \code{"result =:= result"}, \code{"Both(n > 0)"}). More in (\S\ref{sec:spec}).
\end{description}

Without \code{-i}, the auto-aligned bicommand is written to the output file and
printed to stdout, then the process exits.  With \code{-i} the full interactive
rewriting server is started, including the browser UI at \code{/ui}.



% ============================================================
\section{Relational specifications}
\label{sec:spec}
% ============================================================
\code{-rpre}/\code{-rpost} are parsed and scope-checked.

\paragraph{Parsing} A dedicated parser start symbol \code{rformula_top} (in
\code{parser.mly}) and \code{Astutil.parse_rformula_string} parse one
\code{rformula} from a string into the surface AST.

\paragraph{Scope checking.} The formula is type-checked with
\code{Typing.tc_rformula} against a \emph{restricted} \code{bi_tenv} built by
\code{restricted_tenv}: starting from \code{initial_tenv} (only \code{alloc}),
over the program's class table, extended with exactly the method parameters
---and \code{result} for the postcondition only. Any reference outside the
arguments (or \code{result}) therefore fails type-checking with an ``unknown
identifier'' error. This is the requested check that the spec refers only to the
unary method arguments and the result variable; the precondition cannot mention
\code{result}.

\paragraph{Use.} The checked spec is accessed via
\code{/spec}. It informs the \emph{ranking} of synthesised loop guards
(\S\ref{sec:interactive}): the variables it forces to agree (mined from
\code{Agree}/\code{=:=} atoms by \code{agreed_vars}) decide, when two loop guards
differ, whether lockstep is still recommended.

% ============================================================
\section{Default alignment: the sequential alignment}
% ============================================================

The default alignment is computed by \code{Auto.compose_sequentially}. It
 produces the bare, \emph{sequential} bicommand
$(C \mid C')$: it runs all of the left program and then all of the right
program. This is the least aligned adequate product.
It is always projection-valid, and it is the starting point from which the
weaving rewrites introduce synchronisation.



% ============================================================
\section{The rewrite algebra (\texttt{rewrites.ml})}
\label{sec:rewrite}
% ============================================================

A rewrite is a partial, projection-preserving transformation on a bicommand:
\begin{lstlisting}
type rewrite = bicommand -> bicommand option  
\end{lstlisting}

\subsection{Forward weaving (coarse $\to$ fine)}


The rewrites follow the \emph{weaving relation} in ~\cite{banerjee-toplas-2022} (our version of biprograms have been extended with \code{assert}/\code{assume} bicommands). Every
rewrite preserves the projections of the bicommand, so any sequence of rewrites
applied to the default alignment is again an alignment of the same two unary
methods.


\begin{lstlisting}
weave_vardecl : (Var x | Var y in (C|C'))  -->  Var x|y in (C|C')
weave_seq     : (C1;C2 | C1';C2')          -->  (C1|C1') ; (C2|C2')
weave_sync    : (A | A)                    -->  |_ A _|   
weave_if      : (if E .. | if E' ..)       -->  if E|E' 
                                                then (..|..) 
                                                else (..|..)
weave_if4     : (if E .. | if E' ..)       -->  Biif4 with all four branch 
                                                      pairings
weave_while   : (while E .. | while E' ..) -->  while E|E' . <false|false> 
                                                do (..|..)
\end{lstlisting}

\begin{itemize}
  \item \code{weave_seq} is one application of the BiKAT left/right-commute law.
        Applied repeatedly down a right-nested sequence it peels the two programs
        apart one statement at a time.
  \item \code{weave_sync} fires only when the two atomic commands are
        structurally equal (the test used by \code{Annot.eqv_command}).
  \item \code{weave_if} produces the two-way \code{Biif} (lockstep); it is
        projection-valid always but \emph{adequate} only when the guards agree
        --- that becomes a proof obligation. \code{weave_if4} produces the
        four-way \code{Biif4}, which is adequate unconditionally but de-aligns;
        it is offered in \code{/suggest} but never used by \code{auto_align}.
  \item \code{weave_while} generates the \emph{lockstep} loop (both alignment
        guards \code{false}) and carries the unary loop invariants across
        (\S\ref{sec:invariants}).
 \item \code{weave_while_guard ag} build biwhiles with custom alignment guards ag.
\end{itemize}

\subsection{Inverse weaving (fine $\to$ coarse)}

\begin{lstlisting}
unweave_seq : (C1|C1') ; (C2|C2') -->  (C1;C2 | C1';C2')  
unsync      : |_ A _|             -->  (A | A)            
\end{lstlisting}


\subsection{Carrying and suggesting invariants}
\label{sec:invariants}

Weaving a loop must not discard the invariants the unary loops were verified
with. \code{lift_unary_invariants} converts them: an invariant present
(syntactically) on \emph{both} loops becomes \code{Rboth}, a left-only one
\code{Rleft}, a right-only one \code{Rright}; \code{biwhile_spec_of} also pairs
the two frames. This is projection-safe --- \code{projl}/\code{projr} recover
each side's invariants, the other side's projecting to \code{Ftrue} --- and both
\code{weave_while} and \code{weave_while_guard} use it, so weaving a loop no
longer throws the invariants away.

On top of the carried invariants the tool suggests \emph{relational} ones.
\code{coupling_candidates} builds agreement invariants \code{v =:= v} for each
variable occurring in both loops (collected from the guards and bodies by
\code{evars_of_exp} / \code{evars_of_command}). These are \emph{candidates}, not
proven invariants --- a counter that drifts will not agree --- so the user or MCP
keeps the ones the proof needs.

\code{add_invariant rf} adds a relational invariant \code{rf} to a
\code{Biwhile}. The interactive layer feeds it either a chosen candidate or an
arbitrary, MCP-supplied formula (\S\ref{sec:invsuggest}).

\subsection{Focusing}

A rewrite applies at a sub-term selected by a path of child indices:
\begin{lstlisting}
type path = int list   (* e.g. [0;1]; [] = root *)
child_count, get_child, with_child   (* the zipper primitives *)
apply_at  : path -> rewrite -> bicommand -> bicommand option
subterm_at: path -> bicommand -> bicommand option
scope_at  : path -> bicommand -> varbind list * varbind list  (* locals in scope at the focus *)
suggest_at: path -> bicommand -> (string * bicommand) list  (* applicable rewrites + results *)
\end{lstlisting}

\code{scope_at} collects the \code{Bivardecl} bindings passed through on the way
to a focus (split into left/right); with the method parameters it gives the
typing context for an invariant at a loop (\S\ref{sec:invsuggest}).

\code{all_rewrites} is the named registry used by \code{suggest_at}
(\code{weave_vardecl}, \code{weave_seq}, \code{weave_sync}, \code{weave_if},
\code{weave_if4}, \code{weave_while}, \code{unweave_seq}, \code{unsync}).

\subsection{Greedy driver}

\code{auto_align} repeatedly applies the forward weaving rules, descending into
the structure, to turn the sequential default into the finest \emph{lockstep}
alignment the shared structure allows. Statements that do not line up are left
as a \code{Bisplit}. It is only a suggestion; the user refines it.



% ============================================================
\section{Interactive server (\texttt{interactive.ml})}
\label{sec:interactive}
% ============================================================

With \code{-i}, \code{Interactive.run} builds the default alignment, checks the
spec, and serves a stateful API on the chosen port. State is a mutable
\code{current} bicommand plus two stacks: \code{history} (undo) and \code{future}
(redo).

\subsection{HTTP API}

\begin{lstlisting}
GET  /                        help text
GET  /ui                      single-page browser UI
GET  /bicom                   current alignment (text)
GET  /bicom.html              current alignment (line-numbered HTML)
GET  /spec                    relational pre/postcondition
GET  /history                 undo/redo availability and stack depths (JSON)
GET  /rules                   all rule names (incl. "undo","redo") (JSON)
GET  /suggest                 every applicable rewrite, all foci (JSON)
GET  /suggest?path=0.1        applicable rewrites at one focus (JSON)
GET  /invariants?path=P       carried invariants + coupling candidates at P (JSON)
POST /rewrite?rule=R&path=P                 apply rule R at focus P
POST /rewrite?rule=weave_while&guard_left=L&guard_right=R&path=P
                                weave a loop with custom alignment guard <L|R>
POST /rewrite?rule=add_invariant&inv=K&path=P     add coupling candidate K
POST /rewrite?rule=add_invariant&formula=R&path=P add MCP-supplied invariant R
POST /rewrite?rule=undo | rule=redo         undo / redo (rule-style)
POST /undo  /  POST /redo                   undo / redo (verb-style)
POST /auto                    greedily weave toward lockstep
POST /reset                   back to the sequential default
POST /export                  write a .rl bimodule skeleton to the output file
\end{lstlisting}

A \code{path} is a dot-separated list of child indices (\code{0.1}); empty is the
root. Mutating endpoints return the updated alignment as text; \code{409} means
the rule did not apply at the focus, \code{400} a malformed request.

\subsection{Undo / redo}

So that the MCP can undo its rewrites, every successful rewrite \code{snapshot}s
the previous state onto \code{history} and clears \code{future}; \code{/undo}
moves the top of \code{history} to \code{future} and back to \code{current};
\code{/redo} is the mirror image. Undo/redo span \emph{rewrite} steps only:
\code{/auto} and \code{/reset} clear both stacks. The same operations are reached
either as verbs (\code{/undo}, \code{/redo}) or as pseudo-rules through
\code{/rewrite} (\code{rule=undo}, \code{rule=redo}) so a client can drive
everything through one endpoint; \code{/rules} lists them alongside the real
rewrites, and \code{/history} reports availability and depths.

\subsection{Biwhile alignment guards}

Loop weaving with a custom alignment guard is done directly via:
\begin{lstlisting}
POST /rewrite?rule=weave_while&guard_left=L&guard_right=R&path=P
\end{lstlisting}
where \code{L} and \code{R} are arbitrary relational formulas (rformulas) for
the left-side and right-side drain conditions of the alignment guard. Each
side is parsed with \code{parse_rformula_string} and type-checked in the
loop-local \code{bi_tenv} (method parameters, \code{result}, and all locals
in scope at the focus) before the rewrite is applied.

\paragraph{Syntax note.} Guards must be top-level rformulas. The bare
literals \code{True} and \code{False} are unary formula atoms and are not
valid at the rformula level; use \code{Both (true)}, \code{Both (false)},
\code{<| true <]}, or \code{[> false |>} instead. Any rformula involving
\code{Agree}, \code{=:=}, \code{Both}, \code{<| \ldots <]}, or
\code{[> \ldots |>} is accepted provided all variables type-check.

The browser UI exposes the two guard fields as \emph{Left Guard} and
\emph{Right Guard}; leaving both empty applies the lockstep default
(\code{Both (false)} on each side).

\subsection{Biwhile Invariants}
\label{sec:invsuggest}

\code{/invariants?path=P} reports, for a loop focus, the invariants already
carried (\S\ref{sec:invariants}) and the indexed coupling candidates. There are
two ways to add one:
\begin{itemize}
  \item \code{POST /rewrite?rule=add_invariant&inv=K&path=P} --- add candidate
        \code{K} from that list; or
  \item \code{POST /rewrite?rule=add_invariant&formula=R&path=P} --- add an
        arbitrary, caller- (or MCP-)supplied relational invariant \code{R}.
\end{itemize}

The \code{formula} form is what lets an agent contribute its own reasoning ---
e.g.\ a coupling such as \code{i + 1 =:= i} that no \code{v =:= v} candidate can
express. \code{R} is parsed and type-checked \emph{in the loop's local scope}:
\code{scope_at} collects the locals in scope at the focus, and
\code{add_mcp_invariant} builds a \code{bi_tenv} from the method parameters,
\code{result} and those locals, then runs \code{Typing.tc_rformula} against it.
A reference to anything not in scope is rejected with a type error. Both forms
require the loop to be woven (a \code{Biwhile}) first.

\section{HTTP transport and browser UI}
% ============================================================

\subsection{HTTP transport (\texttt{http\_server.ml})}

\code{http_server.ml} is the transport layer; it knows nothing about alignments.
It provides \code{start_dispatch_server}: a generic request-dispatching server
used by the interactive session.  The caller supplies a handler
\code{~meth ~path ~query -> (status, content-type, body)}; the server parses the
request line (\code{parse_request}, \code{query_params}), calls the handler, and
formats the HTTP response (\code{http_response}).  Handler exceptions become
\code{500}.

The interactive API passes all parameters in the query string, so request bodies
are never parsed.  \code{query_params} percent-decodes keys and values
(\code{url_decode}), allowing arguments such as a relational invariant passed to
\code{add_invariant} to contain spaces and punctuation.  The server runs through
\code{run_accept_loop}, which installs SIGINT/SIGTERM handlers and closes the
listening socket for a clean shutdown.

\subsection{Browser UI (\texttt{ui.ml})}

\code{ui.ml} is pure rendering; it contains no socket or HTTP logic.  It
provides two functions:
\begin{itemize}
  \item \code{build_html} --- renders a bicommand as a dark-theme,
        line-numbered HTML page (served at \code{/bicom.html}).
  \item \code{build_interactive_html} --- generates the full single-page
        application served at \code{/ui}.  The page drives the API directly
        from the browser: inspect the current alignment, load suggestions,
        apply rewrites at a focus (filling in path, rule, guards, and
        invariant), undo/redo, auto-align, reset, and export.
\end{itemize}

\code{interactive.ml} calls these two functions when responding to \code{/ui}
and \code{/bicom.html}; neither function is called from \code{http_server.ml}.

% ============================================================
\section{Export}
% ============================================================

\code{/export} writes a \code{.rl} skeleton (\code{bimodule_skeleton}) to the
output file: the current woven bicommand wrapped in a \code{bimodule} stub with
placeholder relational specs (\code{requires true}, \code{ensures true}) and a
\code{TODO} header. The biinterface and method signatures are left for the user
to fill in. This is a first cut --- see \S\ref{sec:open}.

% ============================================================
\section{Pretty-printing}
% ============================================================

\code{Pretty.pp_bicommand} renders bicommands. The \code{Biwhile} case prints the
alignment guards \emph{and} the loop invariants in the header
(\code{while E | E' . P | P' do  invariant {...} ...}); it previously dropped
both, which made synthesised guards and carried/added invariants invisible.

\section{Design decisions and open questions}
\label{sec:open}
% ============================================================

\begin{enumerate}
  \item \textbf{Projection correctness as invariant.} Every rewrite preserves
        $\pi_L(B) \equiv L$ and $\pi_R(B) \equiv R$. Rewrites do \emph{not}
        guarantee fault-freedom/adequacy (e.g.\ \code{weave_if} bets on guard
        agreement); that remains the prover's responsibility, as in the weaving
        discipline.

  \item \textbf{configurable port.} Servers take \code{-port}
        (default 8080)

  \item \textbf{state and undo.} The interactive server holds a single
        \code{current} reference plus \code{history}/\code{future} stacks; focus
        is an explicit path argument rather than a zipper held in state.

  \item \textbf{Custom guards at loop focus.} Relational
        \emph{invariants} of any shape can be supplied and type-checked at the
        loop (\S\ref{sec:invsuggest}) using the loop-local context from
        \code{scope_at}. The same scope-aware typing applies to caller-supplied
        alignment \emph{guards} (\S\ref{sec:guards}): \code{guard_left} and
        \code{guard_right} each accept any well-formed rformula
        (e.g.\ \code{Both (true)}, \code{Agree x}, \code{i =:= i}),
        parsed and type-checked in the loop-local \code{bi_tenv} before the
        rewrite is applied.

  \item \textbf{Spec environment.} The scope check rejects everything but
        arguments/result because the restricted environment omits user predicate
        signatures. Seeding it from the modules' predicate declarations would let
        specs mention user-defined relational predicates.

  \item \textbf{Export fidelity.} The exported skeleton has placeholder method
        signatures and no biinterface. Deriving the biinterface (cf.\
        \code{pretrans/derive_biinterface.ml}) and the real signatures would let
        the result feed straight back into the WhyRel pipeline.

  \item \textbf{Conditional weaving granularity.} \code{weave_if} (two-way) and
        \code{weave_if4} (four-way) are both offered; the two-way form is the
        usual proof target and the only one \code{auto_align} uses.
\end{enumerate}



% ============================================================
\section{Code overview}
\label{sec:code}
% ============================================================

\begin{description}
  \item[\code{align.ml}] is the entry point into the workflow.
  \item[\code{auto.ml}] has sequential composition function.
  \item[\code{rewrites.ml}] the rewrite algebra: weaving/unweaving rules, loop
        guard synthesis, focus paths, and the \code{auto_align} driver.
  \item[\code{interactive.ml}] the stateful session layer: owns the mutable
        \code{current} bicommand, undo/redo stacks, relational-spec checking,
        rewrite dispatch, invariant suggestion, and export.  Calls into
        \code{http_server.ml} for transport and \code{ui.ml} for HTML; contains
        no socket code itself.
  \item[\code{http_server.ml}] the HTTP transport layer: socket accept loop,
        request-line parsing, response formatting, and graceful SIGINT/SIGTERM
        shutdown.  Has no knowledge of alignments or rewrites.
  \item[\code{ui.ml}] pure HTML/JS rendering: the static bicommand page
        (\code{build_html}) and the single-page interactive application
        (\code{build_interactive_html}).  No socket or session logic.
\end{description}

Dependencies flow one way: \code{interactive.ml} uses \code{auto.ml},
\code{align.ml}, \code{rewrites.ml}, \code{typing.ml}, \code{http_server.ml},
and \code{ui.ml}; \code{http_server.ml} and \code{ui.ml} are independent of
each other and of \code{interactive.ml}; \code{align.ml} never refers to
\code{interactive.ml}.
